# STAR 法面试话术 - 多Agent智能会议助手

> 针对5个核心场景，提供完整的 STAR（Situation-Task-Action-Result）回答模板

---

## 场景一：项目整体介绍

**面试官**: "介绍一下你做的这个会议助手项目？"

### S（情景）
在我之前的公司，每周有 20+ 场会议。痛点非常明显：
- 会议纪要整理平均耗时 **2小时/场**，经常拖延
- 待办事项口头分配后 **40% 会被遗忘**
- 管理层想了解会议效率，但没有数据支撑
- 每周在会议事务上消耗 **9+ 小时**

### T（任务）
我负责**从零设计并实现**一套多Agent智能会议助手系统，目标是：
1. 实时语音转文字，自动识别说话人
2. 自动生成结构化会议纪要
3. 自动提取待办并同步到 Jira/飞书
4. 分析会议效率指标
5. 会后自动推送纪要和跟踪提醒

### A（行动）
**架构设计**:
我设计了 5-Agent 的 Pipeline + 并行架构：
- **Transcription Agent**: Qwen3ASR 实时转写 + pyannote 说话人识别
- **Summary Agent**: LLM 结构化摘要，Few-shot + JSON Schema 约束
- **Action Agent**: LLM 提取待办 + Jira/飞书双向同步
- **Insight Agent**: 规则引擎(统计) + LLM(情绪) 混合分析
- **Follow-up Agent**: 汇聚结果，飞书推送，定时提醒

**编排**: 用 LangGraph 做状态图编排，转写串行、摘要/待办/洞察并行、跟进汇聚。

**关键决策**:
- 选择 Qwen3ASR 而非原版 Whisper（速度快70倍 + 精确时间戳）
- 并行Agent写不同State字段避免冲突
- 每个Agent内部try/catch不中断Pipeline

### R（结果）
- 转写准确率 **95%+**，支持中英双语
- 待办提取准确率 **92%**，Jira 同步成功率 **98%**
- 管理者每周节省 **9小时** 会议纪要工作
- 年化节省人力成本约 **110万**
- 系统处理一场1小时会议仅需 **3分钟**

## 场景二：转写准确率优化

**面试官**: "转写准确率95%是怎么达到的？"

### S
初版使用原版 Whisper 直接转写，准确率只有 82%，特别是中文专业术语、说话人重叠时出错多。

### T
将转写准确率从 82% 提升到 95%+。

### A
1. **模型升级**: 从 Whisper base 升级到 Qwen3ASR large-v2（VAD预处理 + 批量推理）
2. **VAD预处理**: 过滤静音段，降低幻觉（幻觉率下降60%）
3. **热词注入**: 将公司内部术语作为 initial_prompt 注入
4. **后处理**: 标点修正 + 数字规范化 + 专业词纠错
5. **说话人识别优化**: pyannote-audio 3.0 + 预注册说话人 embedding

### R
- 准确率: 82% → 95.3%
- 说话人识别准确率: 88%
- 处理速度: 70x 实时速度（比原版快 10 倍）
- 幻觉率: 12% → 4.8%

## 场景三：并行编排设计

**面试官**: "并行执行是怎么实现的？"

### S
最初5个Agent串行执行，处理1小时会议需要8分钟，其中摘要/待办/洞察各需2分钟且互不依赖。

### T
优化执行效率，将总处理时间控制在3分钟以内。

### A
1. **分析依赖关系**: 画出Agent之间的数据依赖图，发现Summary/Action/Insight互相独立
2. **LangGraph编排**: 从Transcription到三个Agent画三条边，框架自动并行
3. **Fan-in汇聚**: 三个Agent都指向Follow-up，等全部完成后再执行
4. **状态隔离**: 并行Agent写入不同字段（summary/actions/insights），无需加锁
5. **Java版**: CompletableFuture.allOf() 等效实现
6. **Go版**: goroutine + sync.WaitGroup 原生并发

### R
- 总处理时间: 8分钟 → 3分钟（减少62.5%）
- 并行阶段从6分钟降到2分钟
- 三种语言都实现了同样的并行效果

## 场景四：Jira/飞书集成

**面试官**: "Jira同步是怎么做的？如何保证可靠性？"

### S
Action Agent 提取出待办后需要自动同步到Jira和飞书，但外部API经常超时或限流。

### T
实现可靠的双向同步，保证98%的同步成功率。

### A
1. **幂等设计**: 基于 meeting_id + task_hash 生成唯一标识，创建前先查询
2. **重试机制**: tenacity 指数退避重试（1s→2s→4s，最多3次）
3. **降级策略**: Jira不可用时先存本地数据库，后台异步重试
4. **批量同步**: 所有待办项批量创建，减少API调用次数
5. **标签追踪**: 每个Issue添加 `meeting-{id}` 标签便于关联查询
6. **监控告警**: 同步失败率超过5%触发告警

### R
- Jira同步成功率: 98%
- 平均同步延迟: 1.2秒/条
- 零重复创建（幂等保证）
- 飞书任务同步成功率: 97%

## 场景五：系统容错设计

**面试官**: "如果某个Agent挂了怎么办？"

### S
上线初期，LLM API偶尔超时导致整个Pipeline失败，用户体验差。

### T
设计Agent级别的容错机制，保证单个Agent失败不影响整体系统。

### A
1. **Agent隔离**: 每个Agent内部try/catch，捕获所有异常
2. **降级结果**: 失败时写入降级结果（如规则引擎替代LLM）
3. **错误记录**: 所有错误追加到state.errors，不中断Pipeline
4. **Follow-up检查**: 最后一个Agent检查errors，在报告中标注失败项
5. **熔断器**: 连续失败3次后触发熔断，直接返回降级结果
6. **重试策略**: 网络超时自动重试，非网络错误不重试

### R
- 系统可用性: 99.5%
- 单Agent失败不影响其他Agent输出
- Pipeline完成率: 100%（即使部分降级）
- 平均恢复时间: < 5秒（熔断后自动恢复）

## 通用回答技巧

### 遇到不确定的问题

> "这部分在实际项目中我们是这样考虑的... 但如果是更大规模的场景，我认为可以..."

### 遇到追问细节

> "具体来说，代码层面是这样实现的... (给出关键代码逻辑)"

### 遇到质疑数据

> "这个数据是在XX规模的测试集上评估的，具体的测试方法是..."

### 遇到没做过的

> "这部分我们目前是用XX方案，但我了解到更好的做法是... 如果有机会我想尝试..."
